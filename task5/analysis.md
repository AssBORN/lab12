# Анализ бизнес-логики расчета стоимости

## Выбранный фрагмент кода (`task3/main.py`)

```python
def calculate_final_price(
    price: float, 
    quantity: int, 
    seller_id: int, 
    customer_type: int, 
    region_id: int, 
    coupon_value: float
) -> Union[float, str]:
    # 1. Расчет базовой стоимости
    total_price = price * quantity
    
    # 2. Применение скидки клиента
    discount_rate = CUSTOMER_DISCOUNTS.get(customer_type, CUSTOMER_DISCOUNTS["default"])
    total_price -= total_price * discount_rate
    
    # 3. Применение купона
    if coupon_value > 0:
        total_price -= coupon_value
    
    # 4. Добавление регионального налога
    tax_rate = REGIONAL_TAXES.get(region_id, REGIONAL_TAXES["default"])
    total_price += total_price * tax_rate
    
    # 5. Вычет комиссии продавца
    commission_rate = SELLER_COMMISSIONS["default"]
    for sellers, rate in SELLER_COMMISSIONS.items():
        if isinstance(sellers, tuple) and seller_id in sellers:
            commission_rate = rate
            break
            
    final_price = total_price * (1 - commission_rate)
    
    if final_price <= 0:
        return "Ошибка"
    
    return round(final_price, 2)
```

## Объяснение работы простым языком

Эта функция вычисляет, сколько в итоге должен заплатить покупатель за товар. Процесс выглядит так:
1. **База**: Сначала считаем простую сумму: `цена × количество`.
2. **Скидка**: Проверяем, какой тип клиента делает покупку (например, VIP или обычный), и вычитаем соответствующий процент скидки.
3. **Купон**: Если у клиента есть купон на определенную сумму, мы просто вычитаем эту сумму из текущего итога.
4. **Налоги**: В зависимости от региона, где находится покупатель, добавляем к сумме региональный налог (процент).
5. **Комиссия**: В конце мы вычитаем комиссию, которую забирает платформа (зависит от того, какой продавец продает товар).
6. **Проверка**: Если в итоге сумма стала нулевой или отрицательной (например, из-за слишком большого купона), функция сообщает об ошибке. В остальном — округляет результат до двух знаков после запятой.

## Предложения по улучшению

1. **Использование `Decimal` вместо `float`**: 
   В финансовых расчетах использование `float` может привести к ошибкам округления (например, `0.1 + 0.2` не всегда равно `0.3`). Для работы с деньгами в Python следует использовать модуль `decimal`.
   
2. **Вынос данных в БД/Конфиг**:
   Сейчас скидки, налоги и комиссии захардкожены в словарях. В реальном приложении эти данные должны храниться в базе данных или внешнем конфигурационном файле, чтобы их можно было менять без перезагрузки и пересборки всего кода.

3. **Замена возврата строки `"Ошибка"` на исключения**:
   Возвращать разные типы данных (`float` или `str`) — плохая практика. Лучше создать кастомное исключение (например, `PriceCalculationError`), которое вызывающий код сможет перехватить и обработать.

4. **Декомпозиция функции**:
   Функция делает слишком много вещей сразу. Её стоит разбить на несколько маленьких функций: `apply_customer_discount()`, `apply_regional_tax()`, `calculate_seller_commission()`. Это упростит тестирование каждой части по отдельности.

5. **Валидация входных данных**:
   Добавить проверку, что цена и количество не могут быть отрицательными еще до начала расчетов, чтобы избежать некорректных промежуточных результатов.
